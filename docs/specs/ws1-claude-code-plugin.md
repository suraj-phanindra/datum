# WS1 — Datum Claude Code plugin + skills (spec)

Status: approved 2026-06-17. This is the frozen contract for WS1; every track codes
against it. See [`docs/ROADMAP.md`](../ROADMAP.md) for where WS1 sits.

## Goal

Package Datum as a self-contained **Claude Code plugin** so a team installs it in one
step (no `npx datumctl init`) and gets the hooks + MCP server + a set of **skills**
that make agents use Datum natively. Ships entirely on the OSS core; no backend.

Install UX:
```
/plugin marketplace add suraj-phanindra/datum
/plugin install datum@datum
```
`datumctl serve` still runs the bus; `datumctl init` remains for manual / non-plugin
setups. The plugin is the recommended path for Claude Code users.

## Layout

The repo doubles as its own marketplace. The plugin is a self-contained subdirectory.

```
.claude-plugin/
  marketplace.json          # repo root: lists the "datum" plugin at path ./plugin
plugin/
  .claude-plugin/
    plugin.json             # manifest (name: datum)
  hooks/
    hooks.json              # PreToolUse/PostToolUse/SessionStart/Stop -> bundled JS
  .mcp.json                 # the datum MCP server (stdio)
  skills/
    coordinate/SKILL.md
    claim/SKILL.md
    sync/SKILL.md
    resolve-fence/SKILL.md
    decide/SKILL.md
  lib/                      # COMMITTED self-contained bundles (built by build:plugin)
    datum-fence.js
    datum-claim.js
    datum-join.js
    datum-guard.js
    datum-mcp.js
  README.md
```

`lib/` (not `dist/`) holds the committed bundles so the repo `.gitignore` `dist/`
rule does not strip them, and so they do not land on PATH (which `bin/` would).
Bundles are regenerated by `npm run build:plugin` and committed as the plugin's
distributable.

Hook commands use exec form: `node` + `["${CLAUDE_PLUGIN_ROOT}/lib/datum-<hook>.js"]`.
The MCP server: `node ${CLAUDE_PLUGIN_ROOT}/lib/datum-mcp.js` with
`env: { DATUM_BUS_URL: "${user_config.bus_url}" }`. `plugin.json` declares a
`userConfig.bus_url` (default `http://127.0.0.1:4317`).

Hook matchers mirror today's installer: SessionStart (datum-join), PostToolUse and
PreToolUse on `Edit|Write|MultiEdit` (datum-claim, datum-fence), Stop (datum-guard).

## Zero-init: datum-join self-seeds state

So the plugin needs no separate `datum init`, enhance `hooks/datum-join.ts`: when
`.datum/state.json` is missing or incomplete, seed it from the existing git-native
helpers before registering the session:
- `session_id`: new uuid
- `human` / `email` / `branch` / `workspace_id`: `cli/lib/git.ts`
  (`gitUserName`, `gitUserEmail`, `currentBranch`, `deriveWorkspaceId`)
- `bus_url`: committed `datum.json` (via the existing config reader) -> `DATUM_BUS_URL`
  env -> default `http://127.0.0.1:4317`
- `claim_files` / `claim_symbols`: empty; `last_synced_version`: 0

This reuses `cli/lib/{git,config,state}.ts` (esbuild inlines them into the bundle).
Solo users get localhost by default; teams get their bus from the committed
`datum.json` (the existing git-native team layer). The seed must be fail-soft (never
block SessionStart) and idempotent (do not clobber an existing valid state).

## MCP additions (server/mcp.ts)

Keep the 4 existing tools. Add 3 (thin proxies over existing bus endpoints):

| Tool | Input | Bus call | Returns |
|---|---|---|---|
| `datum_claim` | `{ files: string[], symbols?: string[], add?: boolean }` | PATCH `/sessions/:id` `{ claim_files, claim_symbols }` (merge if `add`) | `{ claim_files, claim_symbols }` |
| `datum_sync` | `{}` | GET `/version` + GET `/sessions/:id/advisories` + GET `/deltas?since=<last>` | `{ registry_version, advisories, deltas }` |
| `datum_sessions` | `{}` | GET `/sessions` | `{ sessions }` (human, branch, claim, status) |

Same fail-soft `{ warning }` pattern as the existing tools; session id + human from
`.datum/state.json`. Extend `test/mcp.test.ts` to cover the 3 new tools against a
seeded bus.

## The 5 skills

Each is `plugin/skills/<name>/SKILL.md`, namespaced `/datum:<name>`. The `description`
must clearly state WHEN to use it (that drives auto-invocation). Skills drive the
bundled MCP tools (self-contained, no CLI install). No em dashes. No hackathon framing.

| Skill | Auto-invokes when | Drives | Does |
|---|---|---|---|
| `coordinate` | starting work in a Datum workspace (also `/datum:coordinate`) | `datum_sync`, `datum_sessions` | teaches the claim -> sync -> resolve -> decide loop; shows who else is active and current drift; points at the other skills |
| `claim` | beginning a task or when scope changes | `datum_claim` | publishes a rich intent manifest (files + symbols + task) so the arbiter scopes drift accurately |
| `sync` | before editing a shared contract surface (schema/migration, API route, dependency manifest) | `datum_sync` | reviews advisories + deltas since last sync so the fence never has to fire |
| `resolve-fence` | a write was just denied by the Datum fence | `datum_sync`, `datum_my_advisories`, `datum_deltas_since` | systematic self-correction: re-sync, read the mechanical delta + advisory, fix stale references, verify, retry |
| `decide` | renaming a DB column, changing an API shape, or swapping a dependency version | `datum_decide` | records the decision to the ledger so teammates' agents see it (the producer side of preventing drift) |

## Build

Add `scripts/build-plugin.mjs` (or extend `scripts/build.mjs`) and an
`npm run build:plugin` script: esbuild the 4 hooks + `server/mcp.ts` into
`plugin/lib/*.js` with the same options as the main build (platform node, esm,
bundle, target node18, built-ins external, `__DATUM_DIST__`/`__DATUM_ENTRY__`
injection). The committed `plugin/lib/*.js` are the distributable.

## Verification

- `npm test` stays green (incl. the new MCP tool tests and a datum-join self-seed test).
- `npm run build:plugin` produces `plugin/lib/*.js`; CI asserts the committed bundles
  match a fresh build (`build:plugin` then `git diff --exit-code plugin/lib`).
- Structural validation: `plugin.json`, `hooks/hooks.json`, `.mcp.json`, and
  `marketplace.json` parse and match the documented schema; each `SKILL.md` has valid
  frontmatter (name + description) and references only real MCP tools.
- Smoke test: a bundled hook runs against a seeded bus (like `test/installer.test.ts`)
  and the fence denies a stale write through the plugin's `lib/datum-fence.js`.

## Out of scope (WS1)

Hosted/multi-tenant anything (WS2). SSE/streaming MCP tools. Publishing to a central
marketplace registry (install is from this GitHub repo).
